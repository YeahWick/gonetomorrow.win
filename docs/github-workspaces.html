GitHub Workspaces and Branch Management

Introduction

This guide explains the workflow for using GitHub workspaces and managing branches in this project. Understanding this process is crucial for effective collaboration and maintaining a clean, organized codebase.

Setting Up Your Workspace

When contributing to this project, follow these steps to set up your development environment:

1. Fork the Repository

Begin by forking the main repository to your personal GitHub account. This creates your own copy of the project where you can make changes without affecting the original.

Visit the repository homepage and click the "Fork" button in the top-right corner.

2. Clone Your Fork

Clone your forked repository to your local machine:

git clone https://github.com/your-username/gonetomorrow.win.git

Navigate into the project directory:

cd gonetomorrow.win

3. Set Up the Original Repository as a Remote

Add the original repository as an "upstream" remote to keep your fork synchronized:

git remote add upstream https://github.com/YeahWick/gonetomorrow.win.git

Branch Management Workflow

Proper branch management is essential for organized development. Follow this branching strategy:

1. Create a New Branch

Always create a new branch for each feature, bug fix, or documentation update. Use descriptive branch names:

git checkout -b feature/new-button-style

Or for bug fixes:

git checkout -b fix/header-responsive-issue

2. Make Your Changes

Implement your changes, additions, or fixes. Make sure to follow the project's code style and conventions.

Commit your changes with clear, descriptive messages:

git add .
                git commit -m "Add responsive styling to header component"

3. Push to Your Fork

Push your branch to your forked repository:

git push origin feature/new-button-style

The first time you push a new branch, you may need to set the upstream:

git push -u origin feature/new-button-style

Creating a Pull Request

Once your changes are pushed to your fork, you can create a pull request to propose your changes to the main project.

1. Navigate to the Pull Request Page

Go to the original repository on GitHub and click on the "Pull requests" tab.

Click the "New pull request" button.

2. Select Base and Compare Branches

In the pull request creation page:

  • Set the base repository to YeahWick/gonetomorrow.win and the base branch to main
  • Set the head repository to your fork (your-username/gonetomorrow.win) and the compare branch to your feature branch (feature/new-button-style)

This configuration proposes to merge your changes from your feature branch into the main branch of the original repository.

3. Fill Out the Pull Request Template

Provide a clear title and detailed description of your changes:

  • Title: Summarize the change concisely
  • Description: Explain what changes were made and why
  • Related issues: Reference any related issues with #issue-number
  • Test plan: Describe how to verify the changes work as expected

4. Submit the Pull Request

Click "Create pull request" to submit your proposed changes.

Alternatively, you can create a pull request directly from the command line using GitHub CLI:

gh pr create --title "Add new button style" --body "This PR adds the new button styling as per design requirements.\n\n## Test\n- [ ] Verify buttons appear correctly on all pages\n- [ ] Check responsiveness on mobile devices"

Best Practices

1. Keep Commits Focused

Each commit should represent a single logical change. Avoid combining unrelated changes in the same commit.

2. Write Clear Commit Messages

Follow the conventional commit format when possible:

  • feat: for new features
  • fix: for bug fixes
  • docs: for documentation changes
  • style: for formatting changes
  • refactor: for code refactoring
  • test: for adding tests
  • chore: for maintenance tasks

Example: feat(button): add hover animation

3. Sync with Upstream Regularly

Keep your fork updated with changes from the main repository:

git fetch upstream
                git merge upstream/main

Or rebase your branch:

git pull --rebase upstream main

4. Address Feedback Promptly

When reviewers provide feedback on your pull request, address it promptly by making additional commits to your branch. These commits will automatically appear in the pull request.

Troubleshooting Common Issues

1. "No commits between branches" Error

If you encounter this error when creating a PR, it typically means:

  • Your branch is based on an outdated commit
  • You haven't made any commits on your branch yet
  • Your branch is already in sync with the target branch

Solution: Make sure you've committed your changes and that your branch contains commits not present in the main branch.

2. Merge Conflicts

If your branch has conflicts with the main branch, you'll need to resolve them:

git pull upstream main
                # Resolve conflicts in the affected files
                git add .
                git commit -m "Resolve merge conflicts"

Conclusion

Following this workflow ensures smooth collaboration and maintains the quality and stability of the codebase. By using proper branching strategies and pull requests, we can effectively review changes, catch potential issues early, and maintain a clean project history.

Remember to communicate with team members when working on related features, and don't hesitate to ask questions if you're unsure about any part of the process.
