What is Kubie?

Kubie is a more powerful alternative to kubectx and kubens for managing Kubernetes contexts and namespaces. It's designed for users who work with multiple Kubernetes clusters and need to switch between contexts and namespaces frequently.

Unlike traditional Kubernetes context switching tools, Kubie provides shell-level isolation for each context, preventing accidental commands in the wrong cluster.

When you're working with multiple clusters (development, staging, production) and multiple namespaces within each cluster, managing your kubeconfig can become messy. Kubie solves this by providing a clean, organized way to handle context switching.

Why Use Kubie?

If you find yourself in any of these situations, Kubie might be the tool for you:

  • Working with 3+ Kubernetes clusters regularly
  • Managing multiple namespaces within each cluster
  • Frequently switching between development, staging, and production environments
  • Tired of typing long kubectl --context and --namespace flags
  • Wanting to prevent accidental operations in the wrong cluster
  • Needing better organization for your kubeconfig files

Warning: Without proper context management, you might accidentally:

  • Deploy to production when you meant to deploy to staging
  • Delete resources from the wrong cluster
  • Expose sensitive data by working in the wrong namespace

Installation

macOS (Homebrew)

brew install kubie

Linux

# Download the latest release
curl -Lo kubie https://github.com/sbstp/kubie/releases/latest/download/kubie-linux-amd64
chmod +x kubie
sudo mv kubie /usr/local/bin/

Configuration

After installation, you need to configure Kubie to use your kubeconfig:

# Create kubie configuration directory
mkdir -p ~/.config/kubie

# Copy your existing kubeconfig
cp ~/.kube/config ~/.config/kubie/config

Or you can set up multiple kubeconfig files in ~/.config/kubie/ for better organization.

Usage Examples

Example 1: Listing Available Contexts
# List all available contexts
kubie ctx

# List contexts with more details
kubie ctx -l
Example 2: Switching Contexts
# Switch to a specific context
kubie ctx production-cluster

# The shell prompt changes to indicate the current context
# All subsequent kubectl commands will use this context
Example 3: Working with Namespaces
# List namespaces in current context
kubie ns

# Switch to a namespace
kubie ns monitoring

# Now all kubectl commands will use the monitoring namespace
kubectl get pods  # Shows pods in monitoring namespace
Example 4: Context and Namespace Together
# Switch to both context and namespace at once
kubie ctx staging-cluster
kubie ns development

# Or use the combined command
kubie ctx staging-cluster --namespace development
Example 5: Shell Integration
# Kubie modifies your shell prompt to show current context
# Example prompt: [staging-cluster:development] $

# You can also spawn a new shell with a specific context
kubie shell production-cluster --namespace default

# This creates a new shell session isolated to that context

Tips & Best Practices

1. Organize Your Kubeconfigs

Store different environment kubeconfigs in separate files:

~/.config/kube/production.yaml
~/.config/kube/staging.yaml
~/.config/kube/development.yaml

2. Use Shell Aliases

Create handy aliases in your shell configuration:

# Add to ~/.bashrc or ~/.zshrc
alias kprod='kubie ctx production-cluster'
alias kstag='kubie ctx staging-cluster'
alias kdev='kubie ctx development-cluster'
alias kns='kubie ns'

3. Context-Aware Prompt

Kubie automatically updates your shell prompt. Make sure your prompt shows the context clearly to avoid mistakes.

4. Regular Cleanup

Periodically review and remove old contexts you no longer use:

# Check which contexts are in your config
kubectl config get-contexts

# Remove unused contexts
kubectl config delete-context old-cluster-name

5. Scripting with Kubie

When writing scripts, be explicit about context:

#!/bin/bash
# Always specify context in scripts
kubectl --context=production-cluster get nodes
